# 编写技术文档

产出易读、可信、可执行的技术文档。

默认目标不是追求“信息越多越好”，而是在满足文档用途的前提下，用尽可能低的阅读成本提供足够的信息，并保持结构分层清晰。

## 核心原则

每次写文档都遵循以下原则：

1. 从目的出发，不从细节出发。
2. 按“先整体、后细节”的顺序组织内容，不要开头就进入实现细枝末节。
3. 每个章节只回答一个明确问题。
4. 优先渐进展开：先摘要，再补充细节。
5. 只写足以支撑读者下一步动作的内容，不做无效扩写。
6. 明确写出假设、约束、权衡、风险与边界。
7. 使用具体、准确、简洁的语言和标题。
8. 避免细节堆砌、重复解释和大段难扫读的文字。
9. 如果文档用于执行操作，优先保证正确性和可执行性，而不是文风上的“好看”。
10. 如果上下文不完整，做一个合理假设并明确标注，不要把不确定内容伪装成事实。

## 先判断文档的职责

在开始写之前，先从用户请求中识别或保守推断以下信息：

- 文档类型
- 主要读者
- 读者读完后要完成什么事
- 该文档承载的是介绍、决策、评审、实施、执行还是交接
- 如果文档不完整或表述含糊，会带来什么风险
- 本次文档需要覆盖的范围边界

如果这些信息不完整，不要先抛一长串问题。能合理推断的先推断，并在文中简短说明假设后继续写。

## 根据用途控制详略

文档详略由用途决定，不是固定模板一刀切。

### 架构文档

目标是帮助读者理解系统全貌和关键设计判断。

应重点写：
- 背景与范围
- 关键组件与边界
- 职责划分与交互关系
- 核心设计决策与权衡
- 约束、风险、非目标

除非用户明确要求，否则弱化：
- 低层代码逻辑
- 精确配置项
- 过细的接口字段枚举
- 面向执行的逐步操作命令

### 后端设计文档

目标是支撑实现对齐、设计评审和跨模块协作。

应重点写：
- 问题定义与业务背景
- 功能范围与非范围
- 模块职责
- 请求流或数据流
- 数据模型与存储设计
- 接口契约与集成边界
- 异常处理、一致性、性能、安全等关键设计点

仅在影响正确性、协作边界或评审结论时，才展开实现级细节。

### 运维或部署文档

目标是让执行者能安全、稳定地完成操作。

应重点写：
- 前置条件与依赖
- 环境要求
- 明确且有顺序的操作步骤
- 命令、输入、预期输出
- 验证方式与健康检查
- 回滚或恢复路径
- 常见失败场景与排查方法
- 权限、安全、变更窗口等限制条件

不要把关键步骤留成默认常识。

### 使用说明或操作指南

目标是帮助读者顺利完成任务。

应重点写：
- 适用对象
- 开始前需要准备什么
- 按执行顺序展开的步骤
- 示例与预期结果
- 常见错误与修复方法
- 后续还能做什么

优先写清楚“怎么做”，不要被概念性介绍挤占主体。

## 使用分层结构

始终按层次组织内容，不要把摘要、设计点、细节、边角情况混在一起。

推荐分层：

1. 概览
2. 范围与假设
3. 核心方案或核心流程
4. 关键细节与边界情况
5. 风险、权衡或排障信息
6. 待确认问题或后续事项

如果某个章节开始变得臃肿，就按问题、流程阶段、组件、场景拆分，不要在一个标题下堆放不相关细节。

## 默认写作流程

生成或改写文档时，默认按以下顺序推进：

1. 判断文档类型和读者目标。
2. 决定为该目标服务所需的最小章节集。
3. 先写一个短概览，让读者迅速判断这份文档是否与自己相关。
4. 先给出主结构，再补支撑细节。
5. 只补会影响评审、实现、执行或协作的细节。
6. 在结尾前明确写出风险、权衡、假设和未决问题。
7. 收紧措辞，删除重复、空话和泛泛而谈的句子。

## 结构规则

统一遵循以下规则：

- 标题必须传达实际含义，不要用空泛标题。
- 主叙述应保持线性，避免让读者来回跳读。
- 并列项、比较项、前置条件、操作步骤优先用列表表达。
- 只有在比较方案、环境、职责、参数时才优先使用表格。
- 支撑性细节放在主结论之后，不要把关键结论埋在后面。
- 讨论设计决策时，要说明为什么这样选，以及放弃了什么方案。
- 讨论流程时，要写清进入条件、执行步骤、验证方式和结束条件。
- 讨论组件时，要写清职责、输入、输出、依赖和失败行为。

## 表达风格规则

- 简洁，但不能含糊。
- 多用具体名词和动词，少用抽象空泛表达。
- 段落尽量短。
- 删除“需要注意的是”“为了能够”“在这种情况下我们可以”这类无效填充，除非它们确实承载精确信息。
- 不要在概览、正文、结论中重复同一件事，除非重复是为了强调某个操作步骤。
- 不要把原始笔记直接堆成最终文档。
- 如果用户需求很宽泛，先给一个干净骨架，再补最关键的内容。
- 如果文档风险较高，必须补充验证与回滚信息。

## 必做质量检查

在输出前，确认以下检查项：

- 第一部分就能看出这份文档是干什么的。
- 从语气和细节粒度上能看出目标读者是谁。
- 结构顺序符合“概览到细节”的阅读路径。
- 关键结论、关键步骤、关键限制没有埋在大段正文中。
- 详略程度符合文档类型。
- 运维类文档包含前置条件、验证方法和恢复路径。
- 设计类文档包含关键决策、权衡和非目标。
- 文档明确写出假设和开放问题，而不是掩盖不确定性。
- 标题信息量足够，且不重复。
- 读者读完后知道下一步该做什么。

## 文档模板

优先使用最接近的模板，再根据场景调整，不要机械照抄。

### 模板：架构文档

```markdown
# 标题

## 概览
- 这份文档描述什么系统或什么变化
- 为什么需要这份文档
- 本次覆盖什么，不覆盖什么

## 背景
- 当前现状
- 关键约束
- 变更驱动因素

## 目标架构或方案概述
- 主要组件
- 组件职责
- 交互方式
- 边界与依赖

## 关键设计决策与权衡
- 决策 1
- 决策 2
- 被放弃的方案及原因

## 风险与缓解措施
- 风险
- 缓解方式

## 待确认问题
- 问题
```

### 模板：后端设计文档

```markdown
# 标题

## 概览
- 问题定义
- 目标
- 范围

## 需求
- 功能需求
- 非功能需求
- 非目标

## 设计方案
- 主要模块与职责
- 请求流或事件流
- 数据模型与存储
- 外部依赖

## 接口与边界
- 输入输出契约
- API 或事件边界
- 校验与异常处理

## 运行与演进考虑
- 性能
- 安全
- 可观测性
- 迁移或发布

## 风险与开放问题
- 风险
- 开放问题
```

### 模板：运维或部署文档

```markdown
# 标题

## 概览
- 这项操作是做什么的
- 什么时候使用
- 由谁执行

## 前置条件
- 所需权限
- 所需工具
- 所需环境状态

## 操作步骤
1. 步骤
2. 步骤
3. 步骤

## 验证方式
- 检查项
- 预期结果

## 回滚或恢复
- 触发条件
- 恢复步骤

## 故障排查
- 现象
- 可能原因
- 处理动作
```

### 模板：使用说明或操作指南

```markdown
# 标题

## 概览
- 用户可以通过这份文档完成什么
- 适用对象是谁

## 开始前
- 前置条件
- 所需输入

## 操作步骤
1. 步骤
2. 步骤
3. 步骤

## 示例
- 输入
- 输出

## 常见问题
- 问题
- 解决方法

## 后续动作
- 相关后续事项
```

## 如何处理信息不完整

当用户提供的信息不足时：

- 除非缺失信息会直接影响正确性，否则不要先抛很多问题。
- 优先推断最可能的读者和用途。
- 必要时增加一个简短“假设”小节。
- 优先给出可复用、适合评审的结构，不要编造细节。
- 不要捏造精确的实现事实、命令、指标、依赖或环境信息。

## 如何改写已有文档

如果用户给了现有草稿，按这个顺序改：

1. 先修结构和章节顺序。
2. 再删重复和细节杂乱。
3. 再补清目的、范围和读者。
4. 再调整详略，使之符合文档类型。
5. 再补缺失的风险、假设、前置条件和权衡。
6. 最后在不改变技术含义的前提下收紧表述。

## 输出要求

除非用户另有要求：

- 直接输出完整 Markdown 文档。
- 输出应是完整结构，不是散乱笔记。
- 开头部分要短，但信息必须足够。
- 如果存在不确定性，要明确写出假设或开放问题。
- 文档深度必须与文档类型和执行风险匹配。

## 需要避免的反模式

不要：

- 还没解释背景就先进入底层细节。
- 用一个平铺直叙的大列表描述复杂系统。
- 把架构文档、设计文档、运维文档当成同一种东西来写。
- 过度解释显而易见的背景，反而压缩真正有用的信息。
- 用模糊表达掩盖不确定性。
- 用“其他说明”“补充信息”“细节”这类低信息量标题替代明确标题。
- 不考虑读者任务差异，机械套用同一模板。
